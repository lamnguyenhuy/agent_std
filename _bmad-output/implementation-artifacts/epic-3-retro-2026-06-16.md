## What Went Well

- **Contract-first design**: `TranslatorModule` interface (`TranslatorResult { artifacts, compatibilityNotes }`) defined in Story 3.1 before any translator was built. All 3 translators implemented against the same contract with zero contract changes.
- **Translator symmetry**: Claude Code, Cursor, and Windsurf translators followed identical internal patterns (`renderRulesSection`, `renderSkillsSection`, etc.), making review and testing predictable.
- **Compatibility notes isolated**: Story 3.5 cleanly added note rendering without modifying any translator — pure UI addition.
- **Shared `normalizeMarkdownText`**: Single normalization utility used across all translators prevented markdown formatting drift.

## What Could Be Improved

- **`kind: "modified"` hardcoded**: All translators label their artifacts as `"modified"` even when files are net-new (first generation). Should be `"created"` for initial generation. Noticed in review but deferred.
- **Translator error handling**: `translate()` exceptions propagate uncaught to `useMemo`, crashing the entire preview panel. No per-translator error boundary. Deferred.
- **`compatibilityNotes` not rendered in UI**: Story 3.5 compatibility notes only embed text inside artifact content, not rendered via the `compatibilityNotes` field on `TranslatorResult`. The `TranslatorPreview` component has no rendering path for notes — they only appear inline in artifact text. This means the field exists but has no UI consumer.
- **No empty state for translators array**: If `playbook.translators` is somehow empty, the UI renders nothing with no fallback.

## Learning Patterns

- **Template-based preview generation**: Building tool-native config formats from the same playbook proved clean and deterministic. The "one playbook → N outputs" pattern worked well.
- **compatibilityNote inline vs structured**: Embedding notes inside artifact content (e.g., `.cursorrules` comment) made them visible but coupled note text to artifact output. Structured notes would be cleaner but required more UI work.
- **Cross-translator consistency**: Having a single reviewer/implementer for all 3 translators ensured identical patterns. Risk if different people implement different translators.
